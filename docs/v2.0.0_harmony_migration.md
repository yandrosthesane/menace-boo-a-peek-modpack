# v2.0.0 — Direct Il2Cpp Types + Harmony Patching

Replaces the reflection-based OnUpdate polling architecture with compiled Il2Cpp type access
and a Harmony prefix on `AIFaction.OnTurnStart`. Same filtering logic, better timing and
no runtime reflection overhead.

## Problem (v1.2.x)

The v1.2.x architecture had several limitations:

| Issue | Impact |
|-------|--------|
| Reflection-based type access | All field reads, method calls, and list operations went through `GameObj.ReadObj()`, `ReadInt()`, string-based method lookup |
| OnUpdate frame polling for turn detection | Checked `TacticalController.GetCurrentFaction()` every frame, compared against previous value to detect transitions |
| Timing gap | Polling detects the turn change *after* it happens — the AI may have already started thinking before filtering runs |
| Performance overhead | Reflection cache + string lookups every frame, even when nothing changed |
| Il2Cpp list construction via reflection | Building `List<Opponent>` required discovering the generic type at runtime and constructing via `Activator.CreateInstance` |

## Solution — Harmony + Direct Types

### Architecture Change

```
v1.2.x:  OnUpdate() → poll faction → detect change → reflect into AIFaction → filter
v2.0.0:  Harmony Prefix on AIFaction.OnTurnStart → filter (guaranteed before AI thinks)
```

### Direct Il2Cpp Type Access

The `modpack.json` references `Assembly-CSharp`, which provides all game types at compile time:

```csharp
// Before (reflection):
var opponents = aiFaction.ReadObj("m_Opponents");
var count = opponents.ReadInt("_size");

// After (direct):
Il2CppSystem.Collections.Generic.List<Opponent> opponents = aiFaction.m_Opponents;
int count = opponents.Count;
```

Key types used directly:
- `Il2CppMenace.Tactical.AI.AIFaction` — `m_Opponents`, `m_IsAlliedWithPlayer`, `GetIndex()`
- `Il2CppMenace.Tactical.AI.Opponent` — `Actor`, `TTL`, `IsKnown()`
- `Il2CppMenace.Tactical.TacticalManager` — `s_Singleton`, `GetFaction(int)`
- `Il2CppMenace.Tactical.BaseFaction` — base type for faction type-checking via `TryCast<AIFaction>()`
- `Il2CppMenace.Tactical.Actor` — `IsAlive`

### Harmony Patch

```csharp
[HarmonyPatch(typeof(AIFaction), nameof(AIFaction.OnTurnStart))]
static class Patch_AIFaction_OnTurnStart
{
    static void Prefix(AIFaction __instance)
    {
        // Runs before each AI unit thinks — guaranteed timing
        plugin.FilterOpponents(__instance);
    }
}
```

Applied via `_harmony.PatchAll(Assembly)` during `OnInitialize`.

### Key Discovery: OnTurnStart Fires Per-Unit

`AIFaction.OnTurnStart` fires once per AI **unit**, not once per faction. With 15 wildlife
units, the prefix runs 15 times during the Wildlife phase. Each call filters opponents
for that specific unit's faction context.

This is actually better than the v1.2.x per-faction approach — if the game rebuilds
`m_Opponents` between units (it doesn't currently, but could), each unit still gets filtered.

### Faction Discovery (Unchanged Logic, New Implementation)

Same three-way classification as v1.2.0, but using `TryCast<AIFaction>()` instead of
`GameObj.Is()`:

```csharp
var baseFaction = tm.GetFaction(i);
var aiFaction = baseFaction.TryCast<AIFaction>();
if (aiFaction == null)
{
    // Non-AI — check for player units
    var actors = EntitySpawner.ListEntities(i);
    if (actors != null && actors.Length > 0)
        PlayerFactions.Add(i);
    continue;
}
if (aiFaction.m_IsAlliedWithPlayer)
    AlliedAiFactions.Add(i);
else
    HostileAiFactions.Add(i);
```

### FilterOpponents (Same Logic, Direct Types)

```csharp
var filtered = new Il2CppSystem.Collections.Generic.List<Opponent>();
for (int i = 0; i < opponents.Count; i++)
{
    var opponent = opponents[i];
    var actor = opponent.Actor;
    var targetObj = new GameObj(actor.Pointer);

    foreach (var enemy in factionEnemies)
    {
        if (LineOfSight.CanActorSee(enemy, targetObj))
        {
            filtered.Add(opponent);
            break;
        }
    }
}
if (stripped > 0)
    aiFaction.m_Opponents = filtered;
```

## What Did NOT Change

- **Filtering logic** — same binary visible/invisible LOS check per opponent
- **Faction discovery** — same 0-15 probe, same three-way classification
- **No awareness persistence** — still instant amnesia when LOS breaks
- **Settings** — same DebugLogging toggle

## Verified In-Game

Tested over 4 rounds of Tactical play with 15 wildlife units and 4 player units:

```
Round 1: OnTurnStart × 15 — stripped 4 unseen, kept 0 (no wildlife has LOS)
Round 2: OnTurnStart × 15 — kept 4, stripped 0 (wildlife walked into LOS)
         Some units: list swapped (5 removed, 3 kept)
Round 3: OnTurnStart × 15 — kept 8-11, stripped 1-3 (partial visibility as players approach)
Round 4: OnTurnStart × 15 — kept 11, stripped 2 (most visible, some still concealed)
```

Zero crashes, zero exceptions, bridge survived all rounds. Every `OnTurnStart ENTER`
has a matching `EXIT`.

## Bridge Crash Resolution

During testing, the game bridge crashed when using the `move` console command. This was
traced to built-in DevConsole commands calling Il2Cpp methods on the HTTP thread without
`EnqueueMainThread()` — a ModpackLoader bug, not BooAPeek. Fixed `move`, `endturn`, and
`skipai` commands. BooAPeek's Harmony patch runs on the main thread (Unity lifecycle) and
was never the cause.
